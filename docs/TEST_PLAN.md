# E2E — план ручного тестирования

> Итерация 8.4. Проверка сквозных сценариев «фронт ↔ бэк» в dev/staging.

## Предусловия

- [ ] Backend запущен (`server`), БД с миграциями
- [ ] Frontend запущен (`client`, `npm run dev`)
- [ ] JWT-ключи сгенерированы, CORS настроен
- [ ] Тестовый пользователь (или регистрация с нуля)

---

## 2. Создать счёт → расход с QR → проверить баланс

| # | Шаг | Ожидаемый результат | ✓ |
|---|-----|---------------------|---|
| 2.1 | Перейти в «Счета» → создать счёт | `POST /api/accounts` 201, счёт в списке | [ ] |
| 2.2 | Запомнить начальный баланс (0.00) | Отображается в карточке/виджете | [ ] |
| 2.3 | Создать категорию расхода (если нужна) | Категория привязана к счёту | [ ] |
| 2.4 | Новая транзакция → тип «Расход» → сканер QR | Камера/загрузка QR открывается | [ ] |
| 2.5 | Сканировать тестовый QR: `t=20260201T1430&s=125050&fn=72890000&fp=1234567890&i=12345` | Поля: сумма 1250.50, дата, fn/fp/fd заполнены | [ ] |
| 2.6 | Сохранить транзакцию | `POST /api/transactions` 201 | [ ] |
| 2.7 | Проверить баланс счёта | Уменьшился на 1250.50 (или на введённую сумму) | [ ] |
| 2.8 | Список транзакций | Новая запись с корректными полями | [ ] |
| 2.9 | Invalidate balance (React Query) | Баланс обновился без перезагрузки страницы | [ ] |

## Критерии приёмки E2E

- Все сценарии 1–4 пройдены без критических дефектов
- Баланс согласован с суммой транзакций после каждой операции
- Ошибки API отображаются пользователю (toast/форма)
